# Redis 如何实现持久化

Redis 虽然以“快”著称，因为它把数据都放在内存中，但内存数据有个致命弱点：断电即失。为了解决这个问题，Redis
提供了三种主要的持久化机制，将内存数据保存到硬盘上，确保服务重启后数据能恢复。
这三种机制分别是：RDB（快照）、AOF（日志）以及 Redis 4.0 引入的混合持久化。
下面我为你详细拆解这三种方式的工作原理、优缺点及适用场景。

1. RDB (Redis DataBase) —— “拍快照”
   RDB 是 Redis 默认的持久化方式。它的核心思想是在指定的时间间隔内，将内存中的数据集全量快照并写入磁盘，生成一个二进制压缩文件（默认名为
   dump.rdb）。
    * 工作原理：  
      Redis 主进程通过 fork 系统调用创建一个子进程。  
      子进程利用操作系统的写时复制（Copy-On-Write）机制，将内存数据写入临时文件，最后替换旧的 RDB 文件。
      主进程在此期间继续处理客户端请求，几乎不受影响。
    * 触发方式：  
      自动触发：在配置文件中设置规则，例如 save 900 1（900秒内有1次修改）。  
      手动触发：使用 SAVE（阻塞主线程，不推荐）或 BGSAVE（后台异步执行）。

2. AOF (Append Only File) —— “记流水账”
   AOF 机制是为了解决 RDB 数据易丢失的问题而生的。它不再记录数据快照，而是记录每一条写操作命令（如 SET, DEL），以日志的形式追加到文件中。
    * 工作原理：
      每当有写命令执行，Redis 就会将其追加到 AOF 缓冲区。  
      根据策略将缓冲区的内容同步到硬盘。
    * 重写机制：  
      为了解决 AOF 文件无限膨胀的问题，Redis 会定期触发重写（Rewrite），将冗余命令（如连续修改同一个
      Key）合并，只保留恢复当前数据所需的最小命令集。
    * 同步策略（fsync）：
      always：每次写操作都同步到硬盘。最安全，但性能最差。  
      everysec（推荐）：每秒同步一次。平衡了性能与安全，最多丢失 1 秒数据。  
      no：由操作系统决定何时同步。性能最好，但最不安全。
3. 混合持久化 —— “终极方案”
   这是 Redis 4.0 推出的“成年人全都要”方案，结合了 RDB 的快速恢复和 AOF 的数据完整性。
    * 工作原理：
      在进行 AOF 重写时，Redis 不再单纯地重写命令，而是将当前内存数据以 RDB 二进制格式写入 AOF 文件的前半段。  
      将重写期间产生的增量写命令以 AOF 文本格式追加到文件的后半段。
    * 恢复过程：
      重启时，先快速加载前半段的 RDB 数据（基线数据）。  
      再重放后半段的 AOF 日志（增量数据）。

| 机制    | 优点               |
|-------|------------------|
| RDB   | 恢复极快，文件紧凑        |
| AOF   | 数据安全，可读性强        |
| 混合持久化 | 结合 RDB 和 AOF 的优点 |
