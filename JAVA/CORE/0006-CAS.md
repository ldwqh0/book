# CAS

CAS（Compare-And-Swap），中文常译为“比较并交换”，是一种实现并发编程的底层原子指令，也是构建无锁（Lock-Free）数据结构的基石。

## 核心原理

CAS 是一种乐观锁机制。它假设在操作期间没有线程会修改共享数据，只有在真正执行更新时才去检查是否有冲突。
一个 CAS 操作包含三个核心操作数：

* V (内存位置 Value)：需要被更新的共享变量的内存地址。
* A (预期值 Expected)：线程认为变量当前应该持有的值（即操作前的快照）。
* B (新值 New)：线程希望写入的新值。

其执行逻辑是原子性的：  
当且仅当内存位置 V 的当前值等于预期值 A 时，处理器才会原子地将 V 的值更新为 B。否则，不做任何操作。  
整个过程由 CPU 的原子指令（如 x86 架构的 cmpxchg 指令）保证，无法被中断。

## 在 Java 中的应用

Java 程序无法直接操作硬件指令，而是通过 sun.misc.Unsafe 类中的 compareAndSwapInt 等 native 方法来间接调用底层的 CAS
指令。  
我们日常开发中常用的java.util.concurrent.atomic 包下的原子类，就是 CAS 的高级封装。  
例如，AtomicInteger的incrementAndGet() 方法（自增）就是通过一个自旋循环来实现的：

```java
// AtomicInteger.incrementAndGet() 的简化原理
public int incrementAndGet() {
    int v;
    do {
// 1. 读取当前值 (预期值 A)
        v = this.value;
// 2. 计算新值 B
        int next = v + 1;
// 3. 执行 CAS 操作
// 如果 this.value 仍然是 v，则更新为 next，操作成功并退出循环
// 否则，操作失败，循环继续，重新读取、计算、尝试
    } while (!compareAndSwapInt(this, v, next));
    return next;
}
```

## CAS 的优势与问题

### 优势

无锁并发，性能更高：避免了传统锁（如 synchronized）带来的线程阻塞、唤醒以及上下文切换的巨大开销。
适用于低冲突场景：在读多写少、竞争不激烈的场景下，性能表现非常出色。

### 问题与解决方案

CAS 虽然高效，但也存在三个主要问题：

#### ABA 问题

描述：一个变量的值从 A 变为 B，然后又变回 A。此时，一个持有预期值 A 的线程执行 CAS
操作会成功，但它并不知道这个值已经被其他线程修改过，这可能导致逻辑错误。  
解决方案：引入版本号（或称“戳记”）。每次更新变量时，版本号也同步递增。Java 中的 AtomicStampedReference
类就是为了解决此问题而设计的，它将值和版本号捆绑在一起进行比较和更新。

#### 循环时间长，开销大

描述：在高并发场景下，多个线程同时竞争一个 CAS 操作，只有一个能成功，失败的线程会进入自旋（循环重试）状态。如果竞争激烈，长时间的自旋会消耗大量
CPU 资源。  
解决方案：对于高竞争场景，传统的阻塞锁可能更合适。或者使用 LongAdder 等更高级的并发工具，它们通过“分段”思想来减少竞争。

#### 只能保证单个变量的原子操作

描述：CAS 只能对一个共享变量进行原子更新。如果需要同时原子性地更新多个变量，CAS 无法直接实现。  
解决方案：可以将多个变量封装到一个对象中，然后使用 AtomicReference 对这个对象进行 CAS 操作。当然，也可以直接使用锁来保证复合操作的原子性。
