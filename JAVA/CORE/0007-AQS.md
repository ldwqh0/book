# AQS

AQS（AbstractQueuedSynchronizer，抽象队列同步器）是 Java 并发编程（java.util.concurrent 包）的核心基石。
简单来说，AQS 是一个用来构建锁和各种同步组件的底层框架。它帮开发者处理了最复杂的“线程排队、阻塞、唤醒”等逻辑，让开发者只需关注“状态”的管理。
你可以把它想象成一个“交通指挥中心”，它管理着红绿灯（状态）和排队通行的车辆（线程队列）。

## AQS 的两大核心法宝

AQS 之所以能掌控并发，主要依赖两个核心组件：

* 同步状态 (state)
  这是一个 volatile int 类型的变量，相当于“红绿灯”或“计数器”。  
  它的含义由子类决定：  
  在 ReentrantLock 中，state=0 表示锁空闲，state>0 表示锁被占用（值为重入次数）。  
  在 CountDownLatch 中，state 表示倒计时计数器的值。  
  在 Semaphore 中，state 表示当前可用的许可数量。
* 等待队列 (CLH 变体)
  这是一个 FIFO（先进先出）的双向链表。
  当线程抢不到锁（或资源）时，AQS 会将该线程封装成一个节点（Node），加入到这个队列的尾部“排队”。
  当资源释放时，AQS 会负责唤醒队列中的“排头兵”去尝试获取资源。

## AQS 的工作原理

AQS 的设计非常精妙，它使用了模板方法模式。    
AQS 父类做的事：负责“脏活累活”，比如维护队列、线程的阻塞（park）与唤醒（unpark）、线程安全地修改状态（通过 CAS 操作）。  
子类做的事：只需要实现几个特定的方法（如 tryAcquire 和 tryRelease），告诉 AQS “怎样才算获取成功” 和 “怎样才算释放成功”。

### 工作流程简述（以独占锁为例）：

1. 线程调用 acquire() 尝试获取资源。
2. AQS 调用子类的 tryAcquire()。
3. 如果成功，线程直接执行；如果失败，线程被加入等待队列并阻塞。
4. 持有资源的线程调用 release()。
5. AQS 调用子类的 tryRelease() 释放状态，并唤醒队列中的下一个线程。

## AQS 的两种资源共享模式

AQS 支持两种资源访问模式，这也是它强大的原因之一：
工作作原理：模板方法模式

| 模式               | 描述                   | 典型应用                                                |
|------------------|----------------------|-----------------------------------------------------|
| 独占模式 (Exclusive) | 同一时刻只能有一个线程持有资源（排他）。 | ReentrantLock (可重入锁)<br>ReentrantReadWriteLock (写锁) |
| 共享模式 (Shared)    | 同一时刻允许多个线程同时持有资源。    | CountDownLatch (可重入锁)<br> Semaphore (信号量)           |

## 基于 AQS 的常见组件

Java 中许多我们熟悉的并发工具都是 AQS 的“孩子”：

* ReentrantLock：基于 AQS 的独占模式，state 代表锁被持有的次数。
* CountDownLatch：基于 AQS 的共享模式，state 代表计数器数值，减到 0 时唤醒所有等待线程。
* Semaphore：基于 AQS 的共享模式，state 代表剩余许可证数量。
* ReentrantReadWriteLock：结合了独占（写锁）和共享（读锁）模式。

## 总结

AQS 是 Java 并发包的“发动机”。它通过状态 (state) 和 队列 (Queue) 这两个简单的概念，配合 CAS 和
模板方法模式，极其优雅地解决了复杂的线程同步问题。理解了 AQS，你就掌握了 Java 并发编程的“任督二脉”。
