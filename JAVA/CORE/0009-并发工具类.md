# 并发工具类

## 1. CountDownLatch

在 Java 并发编程中，CountDownLatch 是一种同步辅助类，用于协调多个线程之间的执行顺序。它允许一个或多个线程等待，直到在其他线程中执行的一组操作完成。

### 1.1 核心工作机制

CountDownLatch 通过一个计数器来工作，该计数器在初始化时被设定为一个正整数：

* await() 方法：调用此方法的线程会被阻塞，进入等待状态，直到计数器的值变为 0。
* countDown() 方法：每调用一次此方法，计数器的值就会减 1。
* 释放信号：当计数器减至 0 时，所有因调用 await() 而处于等待状态的线程都会被唤醒，继续执行后续操作。

### 1.2 主要特点

* 不可复用：CountDownLatch 是一次性的。一旦计数器达到零，它就不能再被重置或重新使用。如果需要可循环使用的屏障，应考虑使用
  CyclicBarrier。
* 高效解耦：与 Thread.join() 相比，CountDownLatch 更加灵活，因为它不要求等待整个线程结束，只需等待线程执行完特定的逻辑（即调用
  countDown()）即可。
* 内存一致性：根据 Java 内存模型 (JMM)，线程在调用 countDown() 之前的操作，对于在 await() 成功返回后的其他线程是可见的。

### 1.3 典型应用场景

* 并行任务同步：主线程启动 N 个子线程去执行任务（如数据加载、资源初始化），主线程调用 await() 等待所有子任务完成后再汇总结果。
* 最大并行度压力测试：初始化计数器为 1 的“启动门”（Start Gate）。让所有测试线程都在 await() 处等待，由主线程调用一次
  countDown() 同时释放所有线程，以模拟真正的并发冲击。

### 1.4 常用 API 方法

| 方法                                         | 描述                      |
|--------------------------------------------|-------------------------|
| CountDownLatch(int count)                  | 构造函数：创建一个具有给定计数器初始值的实例。 |
| void await()                               | 阻塞当前线程，直到计数器为零或线程被中断。   |
| boolean await(long timeout, TimeUnit unit) | 在限定的时间内等待，若超时则返回 false。 |
| void countDown()                           | 递减计数器。                  |
| long getCount()                            | 获取当前计数器的值，通常用于调试。       |  

### 1.5 代码示例

```java
import java.util.concurrent.CountDownLatch;
import java.util.concurrent.TimeUnit;

public class CountDownLatchDemo {

    static void main(String[] args) throws InterruptedException {
        // 1. 初始化计数器为 3
        CountDownLatch latch = new CountDownLatch(3);

        System.out.println("主线程：正在等待子服务启动...");

        // 2. 模拟 3 个并行任务
        for (int i = 1; i <= 3; i++) {
            final int serviceNum = i;
            new Thread(() -> {
                try {
                    // 模拟耗时操作
                    TimeUnit.SECONDS.sleep(serviceNum);
                    System.out.println("服务 " + serviceNum + " 启动完成");
                } catch (InterruptedException e) {
                    e.printStackTrace();
                } finally {
                    // 3. 关键：每个线程执行完后，计数器减 1
                    latch.countDown();
                }
            }).start();
        }

        // 4. 主线程在此阻塞，直到计数器变为 0
        latch.await();

        System.out.println("主线程：所有服务已就绪，开始运行主逻辑！");
    }

}
```

关键点解析：

* latch.countDown()：通常放在 finally 块中，确保即使子线程任务抛出异常，计数器也能正常递减，防止主线程死等。
* latch.await()：主线程会一直卡在这里，直到 3 个 countDown() 都被执行。
* 单次使用：如果你再次调用 latch.await()，它会立即通过，因为计数器已经归零且无法重置。

## 2. CyclicBarrier

CyclicBarrier 是 Java 并发包中另一个重要的同步辅助类，它的功能是让一组线程到达一个**屏障（Barrier）**
时被阻塞，直到最后一个线程到达，屏障才会打开，所有线程继续执行。  
与 CountDownLatch 最大的不同在于它的可重用性。

### 2.1 核心工作机制

CyclicBarrier 内部维护一个计数器（初始值为线程数）：

* await() 方法：线程调用后表示已到达屏障，随后进入阻塞状态。
* 同步点：当最后一个线程调用 await() 时，计数器归零，屏障打破，所有等待的线程被同时唤醒。
* 自动重置：屏障打开后，计数器会自动恢复到初始值，可以直接开始下一轮。

### 2.2 关键特性

* 可循环利用（Cyclic）：这就是它名字的由来。非常适合分步处理的任务（如多阶段的数据清洗、多轮计算）。
* 屏障操作（Barrier Action）：构造时可以传入一个 Runnable 任务。当屏障打开时（即最后一个线程到达后，但在所有线程恢复执行前），该任务由最后一个到达的线程执行一次。
* 全进全退（All-or-Nothing）：如果其中一个线程因超时、中断或异常离开了屏障，那么屏障就会被认为“损坏”（Broken），其他正在等待的线程也会抛出
  BrokenBarrierException 并退出。

### 2.3 典型应用场景

* 多阶段并行计算：比如一个复杂的数值计算任务分为 3 个阶段，每个阶段都需要所有线程都完成前一阶段的结果后，才能开始下一阶段。
* 数据汇总：利用 Barrier Action，在所有子线程处理完局部数据后，由最后一个线程执行汇总逻辑，然后开启下一轮迭代。

### 2.4 常用 API 方法

| 方法                                                 | 描述                                                  |
|----------------------------------------------------|-----------------------------------------------------|
| CyclicBarrier(int parties)                         | 指定需要相互等待的线程数量。                                      |
| CyclicBarrier(int parties, Runnable barrierAction) | 指定等待数量及屏障打破时的预处理任务。                                 |
| int await()                                        | 线程在此等待。                                             |
| await(timeout, unit)                               | 限时等待：在规定时间内等待。如果超时，屏障会被视为“损坏”，并抛出 TimeoutException。 |
| void reset()                                       | 手动将屏障重置为初始状态。                                       |
| getNumberWaiting()                                 | 查询进度：返回当前已到达屏障并在等待的线程数量。                            |
| getParties()                                       | 查询配置：返回构造时要求的参与线程总数。                                |
| isBroken()                                         | 异常检测：判断屏障是否处于损坏状态（由于超时、中断或 reset 导致）。               |

### 2.5 代码示例

```java
import java.util.concurrent.BrokenBarrierException;
import java.util.concurrent.CyclicBarrier;
import java.util.concurrent.TimeUnit;

public class CyclicBarrierDemo {

    static void main(String[] args) {
        int playerCount = 3;

        // 1. 初始化屏障，当 3 个线程都到达时，触发 Barrier Action（打印“进入下一关”）
        CyclicBarrier barrier = new CyclicBarrier(playerCount, () -> {
            System.out.println("\n=== [系统] 所有玩家已就绪，正在加载下一关卡... ===\n");
        });

        // 2. 创建并启动 3 个玩家线程
        for (int i = 1; i <= playerCount; i++) {
            final int playerID = i;
            new Thread(() -> {
                try {
                    // 第一关
                    simulateAction("玩家 " + playerID + " 正在加载第一关...", 1 + playerID);
                    System.out.println("玩家 " + playerID + " 已到达第一关屏障");
                    barrier.await(); // 第一次等待

                    // 第二关（演示重用性）
                    simulateAction("玩家 " + playerID + " 正在加载第二关...", 1);
                    System.out.println("玩家 " + playerID + " 已到达第二关屏障");
                    barrier.await(); // 第二次等待，屏障自动重置

                } catch (InterruptedException | BrokenBarrierException e) {
                    e.printStackTrace();
                }
            }).start();
        }
    }

    private static void simulateAction(String msg, int seconds) throws InterruptedException {
        System.out.println(msg);
        TimeUnit.SECONDS.sleep(seconds);
    }

}
```

关键点解析：

* Barrier Action：在构造函数中定义的 Lambda 表达式。它在最后一名玩家调用 await() 时触发，且由最后一名到达的线程执行。
* await() 的阻塞与释放：每个线程执行到 await() 都会停住。只有当第 3 个线程也执行到 await()，所有人才会同时被唤醒继续往下跑。
* 循环性：代码中演示了两次 await() 调用。第一次屏障打破后，它会自动重置计数，直接支持第二关的同步。
* 异常处理：注意 BrokenBarrierException。如果某个玩家线程掉线（中断或超时），屏障就会损坏，其他等待的玩家会立即收到异常，防止永久死等。

## 3. Semaphore

Semaphore（信号量） 是 Java 并发包中用于控制同时访问特定资源的线程数量的同步工具。
如果说 CountDownLatch 是“倒计时器”，CyclicBarrier 是“集合点”，那么 Semaphore 更像是一个**“停车场的管理员”**
：它手里握着有限的“许可证”（Permits），只有拿到许可证的线程才能进入，用完后必须归还。

### 3.1 核心工作机制

* 许可证（Permits）：初始化时指定总数（比如 5 个）。
* acquire()：线程尝试获取一个许可证。如果没有剩余，线程会被阻塞，直到有其他线程释放许可证。
* release()：线程执行完任务后释放许可证，将其还回 Semaphore，唤醒正在等待的线程。

### 3.2 主要特点

* 流量控制：常用于限制并发访问的上限（如限流）。
* 非绑定性：释放许可证的线程不一定必须是获取它的那个线程（虽然通常是同一个）。
* 公平性选择：构造时可以指定是否“公平”（Fair）。公平模式下，先排队的线程先获得许可。

### 3.3 典型应用场景

* 资源池限制：如数据库连接池、线程池。
* 接口限流（Rate Limiting）：防止瞬时高并发冲垮下游服务。
* 实现互斥锁：如果初始化 Semaphore(1)，它的功能就类似于 ReentrantLock（但不支持重入）。

### 3.4 常用 API 方法

| 方法                                      | 描述                                                     |
|-----------------------------------------|--------------------------------------------------------|
| acquire()                               | 阻塞获取：获取 1 个许可。如果没有可用许可，当前线程会一直阻塞，直到有许可释放或被中断。          |
| acquire(int permits)                    | 批量获取：一次性获取指定数量的许可。                                     |
| release()                               | 释放许可：释放 1 个许可，将其归还给信号量。                                |
| tryAcquire()                            | 尝试获取 1 个许可。如果当前有可用许可，返回 true；否则立即返回 false，不阻塞。         |
| tryAcquire(long timeout, TimeUnit unit) | 在给定的时间内尝试获取许可。如果在超时前拿到了许可则返回 true，否则返回 false。          |
| acquireUninterruptibly()                | 阻塞获取许可，但不响应中断。即使其他线程调用了该线程的 interrupt()，它也会继续等待直到拿到许可。 |
| drainPermits                            | 一口气拿走当前所有剩余的许可，并返回拿走的数量。常用于“清空”资源或强制关闭某个入口             |
| getQueueLength                          | 返回正在等待获取许可的线程估计数量。                                     |
| hasQueuedThreads                        | 查询是否有线程正在等待获取许可。                                       |

### 3.5 代码示例 模拟数据库连接池限流

```java
import java.util.concurrent.Semaphore;
import java.util.concurrent.TimeUnit;

public class SemaphoreDemo {

    static void main(String[] args) {
        // 1. 初始化信号量，只允许 3 个并发许可
        Semaphore semaphore = new Semaphore(3);

        // 2. 模拟 6 个客户端尝试获取连接
        for (int i = 1; i <= 6; i++) {
            final int clientID = i;
            new Thread(() -> {
                try {
                    System.out.println("客户端 " + clientID + " 正在排队等待连接...");

                    // 3. 获取许可（如果没有空位，会在此阻塞）
                    semaphore.acquire();
                    System.out.println(">>> 客户端 " + clientID + " [成功占用] 一个连接");

                    // 模拟业务处理
                    TimeUnit.SECONDS.sleep(2);

                } catch (InterruptedException e) {
                    e.printStackTrace();
                } finally {
                    // 4. 关键：释放许可，让后续线程进入
                    System.out.println("<<< 客户端 " + clientID + " [释放] 了连接");
                    semaphore.release();
                }
            }).start();
        }
    }

}
```

## 4. Exchanger

Exchanger 是 Java 并发包中一个独特的双向数据交换器。它提供了一个同步点，在这个点上，两个线程可以安全地交换彼此的数据

### 4.1 核心工作流程

* 线程 A 调用 exchange(obj1)，如果线程 B 还没到，线程 A 就会被阻塞。
* 线程 B 到达并调用 exchange(obj2)。
* 瞬间完成：线程 A 拿到 obj2，线程 B 拿到 obj1。
* 两个线程同时被唤醒，继续执行后续逻辑。

### 4.2 关键 API

| 方法                                           | 	描述                                   |
|----------------------------------------------|---------------------------------------|
| V exchange(V x)                              | 阻塞交换：将对象 x 传给对方，并获取对方传回的对象。           |
| V exchange(V x, long timeout, TimeUnit unit) | 限时交换：在指定时间内等不到对方，抛出 TimeoutException。 |

### 4.3 为什么用它？（与 BlockingQueue 的区别）

虽然 BlockingQueue 也能传递数据，但 Exchanger 有两个独特优势：

* 极低开销：它不是简单的单向传递，而是对象重用。通过交换缓冲区，避免了频繁创建和销毁对象的 GC 压力。
* 双向同步：它确保两个线程在这一刻是“同步”的，适合需要两方步调一致的场景。

### 4.4 注意事项

* 成对出现：Exchanger 只能在两个线程间起作用。如果有奇数个线程，最后一个会永远等下去。
* 死锁风险：如果一个线程崩溃或没走到交换点，另一个线程会一直阻塞。建议生产环境务必使用带超时时间的 exchange。

### 4.5 代码示例

```java
import java.util.concurrent.Exchanger;

public class ExchangerExample {

    static void main(String[] args) {
        Exchanger<String> exchanger = new Exchanger<>();

        // 线程 A：生产者（负责填满缓冲区）
        new Thread(() -> {
            try {
                String bufferA = "【数据包 A】";
                System.out.println("生产线程：已填满 " + bufferA + "，等待交换...");
                // 交换：把满的给出去，换回一个空的
                bufferA = exchanger.exchange(bufferA);
                System.out.println("生产线程：换回了 " + bufferA + "，开始下一轮填充。");
            } catch (InterruptedException e) {
                e.printStackTrace();
            }
        }).start();

        // 线程 B：消费者（负责清空缓冲区）
        new Thread(() -> {
            try {
                String bufferB = "【空的缓冲区 B】";
                Thread.sleep(2000); // 模拟处理耗时
                System.out.println("消费线程：处理完旧数据，准备交换...");
                // 交换：把空的给出去，换回一个满的
                bufferB = exchanger.exchange(bufferB);
                System.out.println("消费线程：换回了 " + bufferB + "，开始处理数据。");
            } catch (InterruptedException e) {
                e.printStackTrace();
            }
        }).start();
    }

}
```

> 四个工具的核心对比

| 工具             | 核心类比 | 计数器性质 | 线程关系      | 关键特性          |
|----------------|------|-------|-----------|---------------|
| CountDownLatch | 倒计时器 | 一次性   | 一个线程等多个线程 | 减到 0 时唤醒，无法重置 |
| CyclicBarrier  | 人齐开会 | 可循环   | 多个线程互相等待  | 自动重置，支持屏障任务   |
| Semaphore      | 车位限流 | 可循环   | 线程竞争有限资源  | 控制最大并发数       |
| Exchanger      | 特工接头 | 不涉及   | 两个线程交换数据  | 双向同步，数据交换     |
