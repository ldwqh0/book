# 线程池状态

| 状态         | 接收新任务 | 处理队列任务 | 说明                                               |
|------------|-------|--------|--------------------------------------------------|
| RUNNING    | 是     | 是      | 线程池创建后的初始状态，正常接收并处理任务。                           |
| SHUTDOWN   | 是     | 否      | 调用 shutdown() 后进入。不再接收新任务，但会处理队列中已存在的任务。         |
| STOP       | 否     | 否      | 调用 shutdownNow() 后进入。不接收新任务，丢弃队列任务，并尝试中断正在执行的任务。 |
| TIDYING    | -     | -      | 中间过渡状态。当任务清空且线程数为 0 时进入，准备执行钩子方法。                |
| TERMINATED | -     | -      | 最终状态。terminated() 方法执行完毕，线程池彻底终止。                |

线程池的状态转换是单向的，主要通过调用特定方法或满足特定条件来触发：

* RUNNING → SHUTDOWN  
  触发条件：调用 shutdown() 方法。  
  行为：线程池拒绝新任务，但继续执行队列中剩余的任务，并中断空闲的线程。
* RUNNING / SHUTDOWN → STOP  
  触发条件：调用 shutdownNow() 方法。  
  行为：线程池不仅拒绝新任务，还会尝试中断正在执行的任务，并清空任务队列（返回未执行的任务列表）。
* SHUTDOWN → TIDYING  
  触发条件：任务队列（workQueue）为空 且 线程池中活跃线程数为 0。  
  行为：此时所有任务已处理完毕，准备进入清理阶段。
* STOP → TIDYING
  触发条件：线程池中活跃线程数变为 0。
  行为：因为 STOP 状态会直接丢弃队列任务，所以只需等待运行中的任务结束。
* TIDYING → TERMINATED
  触发条件：terminated() 钩子方法执行完成。
  行为：线程池彻底关闭，资源释放。awaitTermination() 方法会在此时返回。

在实际开发中，我们主要与以下几种状态交互：

* shutdown()：  
  这是推荐的关闭方式，用于优雅停机。它让线程池进入 SHUTDOWN 状态，确保已提交的任务能被完整执行，避免数据不一致。
* shutdownNow()：  
  用于强制关闭。它会尝试立即停止所有任务，适用于紧急情况下需要快速释放资源的场景，但可能会导致数据丢失或状态不一致。
* awaitTermination(long timeout, TimeUnit unit)：  
  通常配合 shutdown() 使用。当前线程阻塞，等待线程池进入 TERMINATED 状态，或者直到超时。这能确保主线程在退出前，线程池内的任务已经处理完毕。
