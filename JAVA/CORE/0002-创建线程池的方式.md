在 Java 中，创建线程池主要有三种方式。但需要特别强调的是，出于生产环境的稳定性和可控性考虑，强烈推荐使用 ThreadPoolExecutor
进行手动创建，而不建议使用 Executors 工具类。  
以下是这三种方式的详细说明：

1. 方式一：使用 Executors 工具类（不推荐）  
   Executors 是一个工厂类，提供了一些静态方法来快速创建常见的线程池。这种方式虽然简便，但因其内部实现存在资源耗尽的风险，在《阿里Java开发手册》等规范中被明确禁止在生产环境中使用。
    * newFixedThreadPool(int nThreads): 创建一个固定大小的线程池。  
      特点：线程数固定，使用无界队列 (LinkedBlockingQueue)。  
      风险：如果任务堆积过快，无界队列会持续增长，最终可能导致内存溢出 (OOM)。
    * newCachedThreadPool(): 创建一个可缓存的线程池。  
      特点：核心线程数为0，最大线程数为 Integer.MAX_VALUE，可根据需要创建新线程。  
      风险：在高并发场景下，可能会创建大量线程，耗尽系统资源（如CPU、内存）。
    * newSingleThreadExecutor(): 创建一个单线程的线程池。  
      特点：保证所有任务按顺序执行。  
      风险：同样使用无界队列，存在 OOM 风险。
    * newScheduledThreadPool(int corePoolSize): 创建一个支持定时和周期性任务执行的线程池。

2. 方式二：使用 ThreadPoolExecutor（推荐）

   这是 Java 原生提供的线程池实现，通过手动配置构造函数参数，可以让你清晰地定义线程池的运行规则，避免资源耗尽的风险，是生产环境中的最佳实践。
   核心参数详解
   ThreadPoolExecutor 的构造函数有7个核心参数，理解它们是配置线程池的关键：
    * corePoolSize 核心线程数，线程池长期维持的最小线程数。 CPU密集型任务可设为CPU核数；I/O密集型任务可设为2倍CPU核数。
    * maximumPoolSize 最大线程数，线程池允许创建的最大线程数量。 CPU密集型任务可设为CPU核数；I/O密集型任务可设为2倍CPU核数。
    * keepAliveTime 非核心线程的空闲存活时间，超时会被回收。 根据系统资源和任务类型调整，避免资源耗尽。
    * unit keepAliveTime 的时间单位。 结合任务间隔设置，如60秒，以便及时回收空闲线程。常用 TimeUnit.SECONDS。
    * workQueue 任务队列，用于存放等待执行的任务。 务必使用有界队列（如 new LinkedBlockingQueue<>(100)），防止任务堆积导致
      OOM。
    * threadFactory 线程工厂，用于创建新线程。 可自定义线程名称、优先级等，便于问题排查。
    * handler 拒绝策略，当队列满且线程数达到最大值时的处理方式。 根据业务需求选择，如 AbortPolicy (抛异常)
      、CallerRunsPolicy (
      调用者线程执行)。

3. 方式三：使用 Spring 的 ThreadPoolTaskExecutor

   如果你在使用 Spring 框架，ThreadPoolTaskExecutor 是对 ThreadPoolExecutor 的封装，提供了更符合 Spring 风格的配置方式，通常在
   XML 或 Java Config 中定义。

```yaml
spring:
  task:
    execution:
      pool:
        core-size: 4
        max-size: 16
        queue-capacity: 1000
        keep-alive: 60s
        allow-core-thread-timeout: true
```
