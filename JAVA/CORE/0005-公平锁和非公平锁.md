# 公平锁和非公平锁

公平锁与非公平锁是 Java 并发编程中，特别是在使用 ReentrantLock 时，一对核心的概念。它们的根本区别在于多个线程竞争锁时，是否遵循“先来后到”的顺序。

## 核心区别

* 公平锁 (Fair Lock): 讲究“先来后到”。线程在获取锁时，会严格按照它们申请锁的顺序来分配。如果锁被占用，新来的线程必须乖乖到队列尾部排队，不能插队。
* 非公平锁 (Nonfair Lock): 允许“插队”。线程在申请锁时，无论队列中是否有其他线程在等待，都会先尝试直接获取锁。如果恰好赶上锁被释放，它就能成功抢到，这相当于插了队。

## 源码实现原理

以 ReentrantLock 为例，其内部通过两个不同的同步器（FairSync 和 NonfairSync）来实现这两种模式。

* 非公平锁的实现
  非公平锁的“非公平”体现在两个关键点上，给了线程两次“插队”的机会：  
  入口抢占：在 lock() 方法入口处，线程会直接尝试一次 CAS 操作来抢占锁，完全不顾及队列中是否有等待的线程。  
  再次抢占：即使第一次抢占失败，进入了 AQS 的等待队列流程，在 tryAcquire 方法中，当发现锁状态为空闲（state=0）时，它会再次尝试
  CAS 抢占。
* 非公平锁的实现
  非公平锁的“非公平”体现在两个关键点上，给了线程两次“插队”的机会：  
  入口抢占：在 lock() 方法入口处，线程会直接尝试一次 CAS 操作来抢占锁，完全不顾及队列中是否有等待的线程。  
  再次抢占：即使第一次抢占失败，进入了 AQS 的等待队列流程，在 tryAcquire 方法中，当发现锁状态为空闲（state=0）时，它会再次尝试
  CAS 抢占。

```java
// 非公平锁的 lock() 方法
void lock() {
    if (compareAndSetState(0, 1)) // 第一次插队尝试
        setExclusiveOwnerThread(Thread.currentThread());
    else
        acquire(1); // 抢占失败，进入队列
}

// 在 tryAcquire 中的 nonfairTryAcquire 方法
boolean nonfairTryAcquire(int acquires) {
// ...
    if (c == 0) {
        if (compareAndSetState(0, acquires)) { // 第二次插队尝试
            setExclusiveOwnerThread(current);
            return true;
        }
    }
// ...
}
```

公平锁的实现则严格遵守 FIFO 顺序。它的核心在于，在尝试获取锁之前，会先检查同步队列中是否有比自己更早到达的线程在等待。  
关键方法 hasQueuedPredecessors()：这个方法会判断队列中是否存在“前辈”线程。只有当队列为空，或者当前线程就是队列中第一个等待的线程时，它才会尝试去获取锁。

```java

// 公平锁的 lock() 方法
void lock() {
    acquire(1); // 直接进入AQS流程，没有预先抢锁
}

// 在 tryAcquire 方法中
private boolean tryAcquire(int acquires) {
    // ...
    if (c == 0) {
        // 关键区别：先检查是否有前辈在排队
        if (!hasQueuedPredecessors() &&
                compareAndSetState(0, acquires)) {
            setExclusiveOwnerThread(current);
            return true;
        }
    }
}
```
