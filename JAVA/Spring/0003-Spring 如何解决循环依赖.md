# Spring 如何解决循环依赖

Spring 主要通过三级缓存机制来解决单例（Singleton）Bean 在字段注入或 Setter 注入时的循环依赖问题。
简单来说，Spring 的核心思路是：提前暴露“半成品”Bean。当一个 Bean 还没完全初始化好（属性还没填充完）时，就先把它的一个引用（早期引用）放到缓存中，让依赖它的其他
Bean 能够拿到这个引用，从而打破循环等待的死锁。

## 三级缓存是什么

Spring 使用三个 Map 来充当缓存，它们各司其职：

1. 一级缓存 (singletonObjects)
   • 存放完全初始化好的、可以直接使用的单例 Bean。这是 Bean 的最终归宿。
2. 二级缓存 (earlySingletonObjects)
   • 存放早期的、未完全初始化的 Bean 实例。这些 Bean 已经实例化，但属性还未填充，也未执行初始化方法。
3. 三级缓存 (singletonFactories)
   • 存放一个 ObjectFactory 工厂对象。这个工厂的作用是按需生成早期 Bean 的引用。这是解决 AOP 代理问题的关键。

## 为什么需要三级缓存？

你可能会问，二级缓存已经能存早期引用了，为什么还需要三级缓存？  
核心原因是为了支持 AOP。  
如果一个 Bean 需要被 AOP 代理（例如使用了 @Transactional 或 @Async 注解），Spring 最终注入的应该是它的代理对象，而不是原始对象。

* 如果只用二级缓存，我们只能存入 Bean 的原始对象。
* 通过三级缓存中的 ObjectFactory，Spring 可以延迟创建代理对象。当其他 Bean 需要这个早期引用时，会通过工厂的 getObject()
  方法来获取。这个方法内部会判断是否需要创建 AOP 代理，从而确保注入的是正确的代理对象，保证了 Bean 的一致性。

## 解决循环依赖的完整流程

让我们以两个单例 Bean A 和 B 互相依赖为例（A 依赖 B，B 依赖 A），看看 Spring 是如何一步步解决的：

1. 创建 Bean A
    * Spring 开始创建 Bean A，通过反射实例化 A，得到一个原始对象（此时 A 的属性 b 还是 null）。
    * 关键一步：Spring 会立即将一个能创建 A 早期引用的 ObjectFactory 放入三级缓存。
    * 接着，Spring 开始为 A 填充属性，发现它依赖 Bean B。
2. 创建 Bean B
    * 由于 A 依赖 B，Spring 会暂停 A 的创建，转而去创建 Bean B。
    * 同样地，Spring 实例化 B（此时 B 的属性 a 还是 null），并将 B 的 ObjectFactory 也放入三级缓存。
    * 然后，Spring 开始为 B 填充属性，发现它依赖 Bean A。
3. 解决循环依赖
    * B 需要 A，于是 Spring 开始查找 A。
    * 一级缓存：没找到（A 还没创建完）。
    * 二级缓存：没找到。
    * 三级缓存：找到了 A 的 ObjectFactory！Spring 调用这个工厂的 getObject() 方法，获取到 A 的早期引用（如果需要
      AOP，这里得到的就是代理对象）。
    * 然后，Spring 将这个 A 的早期引用从三级缓存移除，并放入二级缓存。
    * 最后，将这个 A 的早期引用注入给 Bean B，B 的属性 a 赋值成功。
4. 完成 Bean B 的创建
    * B 的属性填充完毕，接着执行初始化方法（如 @PostConstruct）。
    * B 完全初始化完成，被放入一级缓存，同时清理掉它在二、三级缓存中的数据。
    * 创建好的 Bean B 被返回，用于填充 A 的属性。
5. 完成 Bean A 的创建
    * Spring 回到 A 的创建流程，将刚刚创建好的 Bean B 注入给 A，A 的属性 b 赋值成功。
    * A 执行初始化方法，完全初始化完成。
    * A 被放入一级缓存，清理掉二、三级缓存中的数据。
      至此，A 和 B 都成功创建并互相注入了对方，循环依赖问题被完美解决。

## Spring 无法解决的场景

需要注意的是，Spring 的三级缓存机制并不能解决所有循环依赖问题。

### 构造器注入导致的循环依赖

如果 A 和 B 都通过构造器来注入对方，Spring 将无法解决。

* 原因：构造器注入发生在 Bean 实例化阶段。在实例化 A 之前，Spring 就需要先创建一个 B 的实例作为参数。但在实例化 B
  之前，又需要一个 A 的实例。这就形成了一个无法打破的死锁，因为 Bean 还没实例化，就无法放入三级缓存来提前暴露引用。
  解决方案：
* 首选：重构代码，通过引入第三方 Bean 或接口来打破循环依赖，这是最根本的解决方式。
* 次选：在其中一个构造器参数上使用 @Lazy 注解。这会注入一个该 Bean 的代理对象，实际的 Bean 会在第一次被使用时才创建，从而延迟了依赖的加载。

### 原型（Prototype）作用域的循环依赖

对于非单例的 Bean（如 @Scope("prototype")），Spring 不会对其进行缓存，因此也无法解决它们之间的循环依赖，会直接抛出异常。
