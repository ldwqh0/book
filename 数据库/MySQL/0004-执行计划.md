# 执行计划

EXPLAIN 是 MySQL 中用于分析 SQL 语句执行计划的核心工具，堪称 SQL 性能调优的“X光机”。它不会真正执行你的
SQL，而是模拟优化器的决策过程，告诉你 MySQL 准备如何执行这条查询，例如是否使用索引、扫描多少行、表连接顺序等。

## 基本用法

用法非常简单，只需在 SELECT 语句前加上 EXPLAIN 关键字即可。

```Sql
EXPLAIN
SELECT *
FROM users
WHERE id = 1;
```

对于 MySQL 8.0 及以上版本，还可以使用 EXPLAIN ANALYZE，它会真正执行 SQL 语句，并提供更详细的实际运行时间和成本信息，是更强大的分析工具。

```sql
EXPLAIN
ANALYZE
SELECT *
FROM users
WHERE id = 1;
```

## 字段核心含义

| 字段            | 核心含义                                                 |
|---------------|------------------------------------------------------|
| id            | SQL 语句的执行顺序标识。数字越大，优先级越高，越先执行。                       |
| select_type   | 查询类型，如 SIMPLE (简单查询)、PRIMARY (主查询)、SUBQUERY (子查询) 等。 |
| table         | 当前行显示的是哪张表的信息。                                       |
| type          | 最重要的字段之一，表示连接类型或访问类型，直接反映查询效率。                       |
| possible_keys | 指出 MySQL 可能会使用哪些索引来查找行。                              |
| key           | 关键字段，显示 MySQL 实际决定使用的索引。如果为 NULL，则表示没有使用索引。          |
| rows          | MySQL 预估需要扫描的行数。数值越小越好。                              |
| Extra         | 包含额外的重要信息，如是否使用了覆盖索引、是否进行了文件排序等。                     |

### type (访问类型) - 性能的晴雨表

* type 字段是判断 SQL 性能好坏的首要指标，其值从优到差依次是：
* system > const > eq_ref > ref > range > index > ALL
* system / const: 最优级别。查询结果最多只有一行，效率极高。例如通过主键或唯一索引进行等值查询。
* eq_ref: 常见于多表 JOIN 查询，使用主键或唯一索引进行关联，效率很高。
* ref: 使用普通（非唯一）索引进行查找，可能会找到多行。
* range: 索引范围扫描，常见于使用 >, <, BETWEEN, IN 等条件的查询。
* index: 全索引扫描，遍历整个索引树。虽然比全表扫描快，但仍有优化空间。
* ALL: 最差级别，全表扫描。MySQL 会逐行扫描整个表，在数据量大时性能极差，是优化的重点对象。

一般来说，至少要保证查询达到 range 级别，最好能达到 ref 或更高。

### key (实际使用的索引)

这个字段告诉你 MySQL 最终选择了哪个索引。  
如果 key 为 NULL，说明查询没有使用任何索引，这通常是性能问题的根源。  
你需要对比 possible_keys 和 key，如果 possible_keys 有值但 key 为 NULL，说明优化器认为使用索引反而更慢，或者 SQL
写法导致索引失效。

### Extra (额外信息) - 优化线索库

* Extra 字段提供了许多关于查询执行的细节，其中几个值需要特别关注：
* Using index: 非常好的信号。表示使用了“覆盖索引”，即查询的列都包含在索引中，无需“回表”查询完整数据行，性能极佳。
* Using where: 表示 MySQL 在存储引擎层检索行之后，还需要在 Server 层使用 WHERE 条件进行过滤。
* Using filesort: 需要警惕。表示 MySQL 无法利用索引来完成排序，而需要在内存或磁盘中进行额外的排序操作，性能开销较大。
* Using temporary: 性能杀手。表示 MySQL 需要创建一张临时表来存储中间结果，常见于 GROUP BY 和 ORDER BY 操作，应尽量避免。
